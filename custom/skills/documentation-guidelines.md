# 문서 작성 가이드

문서를 작성하거나 수정할 때 참고하는 가이드입니다.

## 트리거 조건

다음 상황에서 이 가이드를 참고합니다:

**자동 트리거 (사용자 요청 없이도 문서 작성):**
- 새로운 도구/라이브러리 환경을 세팅한 경우
- 기존 도구의 환경 세팅을 변경한 경우
- 3개 이상의 메시지로 트러블슈팅을 진행하여 문제를 해결한 경우
- 프로젝트에 새로운 기능/시스템을 추가한 경우

**사용자 요청 트리거:**
- "문서 작성해줘", "문서화해줘", "docs에 추가해줘"
- "README 작성해줘", "가이드 만들어줘"
- "정리해서 문서로 남겨줘"

---

## 파일명 규칙

### 필수: kebab-case 사용

```
✅ 올바른 예시:
testing.md
tech-stack.md
api-guide.md
troubleshooting-auth.md
setup-guide.md

❌ 잘못된 예시:
TESTING.md        (UPPERCASE)
TechStack.md      (PascalCase)
api_guide.md      (snake_case)
setupGuide.md     (camelCase)
```

### 파일명 작성 원칙

1. **소문자만 사용**: 대문자 사용 금지
2. **단어 구분은 하이픈(-)**: 언더스코어(_) 사용 금지
3. **의미 있는 이름**: 내용을 유추할 수 있는 이름 사용
4. **영문 사용**: 한글 파일명 사용 금지

## 문서 저장 위치

### 프로젝트 문서 (docs/)

프로젝트별 기술 문서는 해당 프로젝트의 `docs/` 폴더에 저장합니다.

```
project/
└── docs/
    ├── tech-stack.md       # 기술 스택 설명
    ├── testing.md          # 테스트 가이드
    ├── api-guide.md        # API 사용 가이드
    └── deployment.md       # 배포 가이드
```

### 전역 skills (~/.claude/skills/)

여러 프로젝트에서 재사용 가능한 가이드는 전역 skills에 저장합니다.

```
~/.claude/skills/
├── documentation-guidelines.md   # 문서 작성 가이드 (이 파일)
├── frontend-code-principles.md   # 프론트엔드 코드 원칙
└── troubleshooting/              # 트러블슈팅 모음
    └── vitest-esm-error.md
```

## 문서 유형별 가이드

### 1. 기술 문서 (Tech Docs)

환경 세팅, 기술 스택, 아키텍처 설명 문서입니다.

**파일명 예시:** `tech-stack.md`, `architecture.md`, `testing.md`

**필수 포함 내용:**
- 개요 (무엇을, 왜)
- 기술 스택 테이블
- 설정 파일 설명
- 사용 명령어
- 참고 문서 링크

**템플릿:**
```markdown
# [제목]

## 개요
[이 문서가 설명하는 내용 1-2문장]

---

## 기술 스택

| 기술 | 버전 | 역할 |
|------|------|------|
| ... | ... | ... |

---

## 설정

### [설정 파일명]
```[언어]
[설정 내용]
```

---

## 사용 방법

```bash
[명령어]
```

---

## 참고 문서
- [링크](URL)
```

### 2. 트러블슈팅 문서

실제 문제 해결 과정을 기록한 문서입니다.

**파일명 예시:** `troubleshooting-auth.md`, `fix-esm-error.md`

**필수 포함 내용:**
- 문제 상황 (에러 메시지 포함)
- 원인 분석
- 해결 방법
- 예방 방법 (선택)

**템플릿:**
```markdown
# [문제 제목]

## 문제 상황

[에러 메시지 또는 증상]

```
[에러 로그]
```

## 원인

[왜 이 문제가 발생했는지]

## 해결 방법

[단계별 해결 과정]

## 참고
- [관련 이슈/문서 링크]
```

### 3. 가이드 문서

사용 방법, 워크플로우를 설명하는 문서입니다.

**파일명 예시:** `api-guide.md`, `deployment-guide.md`, `contribution-guide.md`

**필수 포함 내용:**
- 목적
- 사전 요구사항
- 단계별 가이드
- 예제 코드
- FAQ (선택)

## 작성 원칙

### 내용 원칙

1. **간결하게**: 800라인 이내 권장, 최대 1000라인
2. **중복 제거**: 작성 후 중복 내용 검토
3. **코드 예제 포함**: 복사해서 바로 쓸 수 있는 예제
4. **민감 정보 제외**: API Key, Secret은 placeholder 사용
   ```
   ✅ GEMINI_API_KEY=your-api-key-here
   ❌ GEMINI_API_KEY=AIzaSyB...실제키...
   ```

### 포맷 원칙

1. **제목 계층**: H1은 문서당 1개, H2로 섹션 구분
2. **구분선 사용**: 주요 섹션 사이에 `---` 사용
3. **테이블 활용**: 비교, 목록은 테이블로 정리
4. **코드 블록**: 언어 명시 (```typescript, ```bash 등)

### 트러블슈팅 문서 특별 원칙

- **실제 경험만**: 가상의 시나리오로 작성 금지
- **에러 메시지 포함**: 검색 가능하도록 실제 에러 메시지 기록
- **해결된 방법만**: 시도했으나 실패한 방법은 간략히만 언급

## 문서 디렉토리 구조

관련 문서가 3개 이상이면 하위 디렉토리로 그룹화합니다.

```
docs/
├── testing.md                    # 단독 문서
├── tech-stack.md
└── api/                          # 관련 문서 그룹
    ├── overview.md
    ├── authentication.md
    └── endpoints.md
```
