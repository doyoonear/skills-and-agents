# Solutions Suggest - 해결 사례 자동 제안

세션 중 해결한 문제들을 분석하여 `docs/solutions/` 문서로 기록할 가치가 있는 후보를 제안한다.

## 트리거 조건

이 스킬은 다음 상황에서 사용한다:

1. **명시적 호출**: 사용자가 `/solutions-suggest` 또는 관련 키워드 사용
2. **wrap 스킬 연계**: `/wrap` 완료 후 자동 제안 (사용자 선택)
3. **리뷰 에이전트 완료 후**: compound-engineering 리뷰 에이전트가 문제를 발견하고 수정한 뒤

## 워크플로우

### 1. 세션 컨텍스트 수집

현재 대화에서 다음 정보를 파악한다:

- 이번 세션에서 **발생한 문제와 그 해결 과정**
- **코드 리뷰에서 지적되어 수정한 사항**
- **디버깅에 시간이 걸린 이슈**
- **설계 결정에서 한 번 잘못 갔다가 되돌아온 경우**

### 2. 후보 선별 — 기록할 가치 판단

모든 해결 사례를 기록하는 것이 아니라, **아래 기준에 부합하는 것만** 후보로 선별한다.

#### 기록해야 하는 것 (INCLUDE)

| 기준 | 예시 |
|------|------|
| **코드만 봐서는 "왜"를 알 수 없는 결정** | implicit flow → PKCE 전환 이유 |
| **같은 실수가 반복될 가능성이 높은 패턴** | DELETE에 toggle 사용 → 멱등성 위반 |
| **프레임워크/라이브러리의 비직관적 동작** | Supabase scope 설정이 원격에 남는 문제 |
| **해결에 30분 이상 소요된 디버깅** | 증상과 원인이 먼 경우 |
| **환경/설정 관련 함정** | 로컬 vs 프로덕션 차이로 인한 버그 |

#### 기록하지 않는 것 (EXCLUDE)

| 기준 | 예시 |
|------|------|
| **코드 diff로 충분히 전달되는 변경** | 변수명 변경, import 추가 |
| **단순 타입 에러 수정** | `any` → 명시적 타입 |
| **일회성 설정/환경 문제** | 특정 머신에서만 발생하는 이슈 |
| **이미 공식 문서에 잘 설명된 것** | 프레임워크 기본 사용법 |
| **새 기능 구현 자체** | 기능의 존재는 코드가 증명 |

### 3. 후보 제안

선별된 후보를 사용자에게 표 형태로 제안한다.

```
📋 Solutions 문서 후보

| # | 제목 | 기록 이유 | 추천도 |
|---|------|-----------|--------|
| 1 | DELETE 핸들러 멱등성 위반 | toggle 패턴의 HTTP 시맨틱 충돌, 반복 가능 | ★★★ |
| 2 | Supabase scope 원격 설정 잔존 | 코드에 없는 설정이 동작에 영향, 디버깅 난이도 높음 | ★★★ |
| 3 | ... | ... | ★★☆ |

작성할 번호를 선택하세요 (예: 1,2) 또는 "전체"/"건너뛰기"
```

**추천도 기준:**
- ★★★: 반복 가능성 높음 + 원인 파악이 어려움
- ★★☆: 반복 가능성 있음 OR 원인 파악이 어려움
- ★☆☆: 기록하면 좋지만 필수는 아님

### 4. 선택된 후보 문서 작성

사용자가 선택한 후보에 대해 `docs/solutions/` 에 문서를 작성한다.

**파일명 형식:** `{YYYY-MM-DD}-{주제-kebab-case}.md`

**문서 구조:**

```markdown
# {문제를 한 문장으로 요약한 제목}

## 문제 요약

{무엇이 잘못되었는지 2-3문장}

## 관찰된 증상

{사용자/개발자가 실제로 본 현상. 에러 메시지, 비정상 동작 등}

## 원인 분석

{왜 이런 문제가 발생했는지. 코드 레벨 설명 포함}

## 해결 방법

{실제로 적용한 해결책. 코드 스니펫 포함}

## 수정한 파일

- {파일 경로 목록}

## 재발 방지 메모

- {같은 상황에서 기억해야 할 원칙 3-4줄}
```

**작성 원칙:**
- "문제 → 증상 → 원인 → 해결" 순서를 반드시 지킨다
- 코드 스니펫은 핵심 부분만 포함 (전체 파일 복사 금지)
- "재발 방지 메모"는 **다음에 같은 상황을 만났을 때 바로 적용할 수 있는 규칙** 형태로 작성

### 5. 완료 보고

- 생성된 파일 경로
- 각 문서의 한 줄 요약

## wrap 스킬 연계

`/wrap` 스킬의 "4. 조건부 Handoff" 단계 이후에 이 스킬을 연계할 수 있다.

```
/wrap 완료 → "이번 세션에서 해결한 문제 중 solutions 문서로 남길 것이 있는지 확인할까요?" → 사용자 동의 시 실행
```

연계는 자동이 아니라 **사용자 동의 후** 실행한다.

## 주의사항

- solutions 문서는 **커밋에 포함**한다 (handoff와 다름). 팀 전체가 참고할 수 있어야 하므로.
- 기존 `docs/solutions/` 문서와 중복되는 내용은 작성하지 않는다. 작성 전 기존 파일 목록을 확인한다.
- `learnings-researcher` 에이전트가 이 문서를 frontmatter 없이도 검색할 수 있도록, 제목과 섹션 구조를 일관되게 유지한다.
