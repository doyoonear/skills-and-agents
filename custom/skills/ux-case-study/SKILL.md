# UX Case Study (Devouring Details)

Devouring Details 강의의 한 장을 입력으로 받아, 본문에 없는 **새 패턴 + 캡처 가능한 사례**로 카드뉴스 리서치 문서를 만드는 스킬.

## 트리거

- 슬래시: `/ux-case-study [장 번호]`
- 한국어:
  - "Devouring Details N장 케이스 스터디"
  - "N번째 장 새로운 패턴 찾아줘"
  - "UX 케이스 스터디 만들어줘"
  - "물리적인 움직임 새 사례"

장 번호가 명시되지 않으면 사용자에게 묻는다.

## 입력 / 출력

### 입력
- **장 번호** (예: 4)
- (선택) 발굴할 패턴 개수 (기본 5~7개)

### 출력
- **파일 경로**: `/Users/doyoonlee/ObsidianVault/Areas/Study/UX 엔지니어링 스터디/Devouring Details/{N}. {원본 제목}_리서치.md`
- 기존에 같은 파일이 있으면 내용을 확인한 뒤 **덮어쓸지 사용자에게 확인**.

## Workflow

### Step 1: 본문 읽기
- 경로: `/Users/doyoonlee/ObsidianVault/Areas/Study/UX 엔지니어링 스터디/Devouring Details/`
- 입력된 장 번호로 시작하는 파일을 Glob으로 찾아 본문(`_리서치` 미포함) 파일을 Read.
- 본문의 H2 헤더, 인용문, NOTE 박스를 모두 읽어 7~N개의 sub-topic을 파악.

### Step 2: 상위 주제(Meta Theme) 정의
- 본문이 다루는 7개 sub-topic의 **공통 추상 개념**을 한 문장으로 정의.
- 예: 4장 → "디지털 인터랙션이 현실 세계의 물리 법칙을 시뮬레이션할 때 사용자는 인터페이스를 자연스럽게 인식한다."
- 이 메타 주제가 새 패턴 도출의 **기준점**이 된다.

### Step 3: 새 패턴 후보 도출
- 본문 sub-topic을 그대로 복제하지 않는다.
- 메타 주제는 같지만 **다른 메타포**를 사용하는 패턴을 5~7개 도출.
- 예 (4장):
  - 본문: 스프링/관성/러버밴딩/projected position
  - 새 패턴: Magnetic Snap / Inertial Cursor / Squash & Stretch / Liquid Morph / Gravity Drop / Parallax / Inertial Throw
- 각 패턴에 한 줄 정의를 붙이고, "본문과 어떻게 다른가"를 명시.

### Step 4: 각 패턴별 사례 A/B 발굴
- 각 패턴마다 **2개의 실제 사례** 발굴.
- **선정 기준**:
  - 공개된 웹페이지의 시각 자료가 캡처 가능하거나
  - 무료 계정으로 가입 후 화면 녹화 가능하거나
  - 운영체제/앱에 기본 탑재되어 사용자가 직접 체험 가능
- **검증 (URL 접근만 — 3장 수준)**:
  - WebFetch로 URL이 살아있고 페이지 본문이 존재하는지 확인.
  - 페이지가 SPA여서 HTML이 비어 보여도, URL이 200 응답이고 잘 알려진 사례면 채택.
  - 깊은 인터랙션 검증은 사용자 캡처 단계로 위임.

### Step 5: 마크다운 작성
다음 구조를 정확히 따른다 (3장/4장 새로운 패턴 리서치와 동일):

```markdown
# {장 제목} — 새로운 패턴별 사례 리서치

> Devouring Details 강의 N번째 원칙 "{영문 제목}"의 **상위 주제를 표현하는, 본문에 등장하지 않은 새로운 패턴**과 그 사례를 정리합니다.
> 카드뉴스 제작을 위해 **직접 캡처 가능하거나 무료 계정으로 녹화 가능한 사례**를 우선 선정했습니다.

> 본문 주제를 그대로 따라간 사례는 [[N. {본문 파일명}_리서치|기존 리서치 문서]]를 참조.

---

## 🎯 N장의 상위 주제 (Meta Theme)

> **"{메타 주제 한 문장}"**

본문에서 다루는 N가지 sub-topic은 모두 이 메타 주제의 한 측면이다. 아래 N가지 새 패턴은 본문에 등장하지 않지만 같은 메타 주제를 표현한다.

---

## 1. {새 패턴 이름} — {짧은 영문/한글 설명}

> 새 패턴 정의: {정의 + 본문과의 차이}

### 사례 A: {제품/회사명} — {핵심 인터랙션}
- **확인 방법**: [{링크 텍스트}]({URL}) 에서 {구체적 행동}하며 캡처/녹화
- **설명**: {2~3문장. 인터랙션이 어떤 물리/메타 주제를 표현하는지}
- **캡처 포인트**: {녹화/스크린샷할 정확한 순간}

### 사례 B: ... (동일 형식)

(패턴 5~7개 반복)

---

## 참고 자료

- [...](URL)
```

### Step 6: 검증
- Obsidian 마크다운 규칙(`-` 뒤 공백) 준수 확인.
- 모든 사례에 URL이 명시되어 있는지 확인.
- "본문에 등장하지 않는" 패턴인지 자체 비판으로 한 번 더 점검.

### Step 7: 사용자 보고
- 작성한 파일 절대경로 출력.
- 발굴한 새 패턴 개수와 이름 목록 출력.
- 사용자에게 **다음 단계** 안내: "각 사례를 직접 캡처/녹화한 뒤 카드뉴스 제작으로 진행하세요."

## 작성 원칙

1. **본문 주제 복제 금지** — 같은 메타포(스프링·바운스·러버밴딩 등)를 다시 사용하지 않는다.
2. **캡처 가능성이 사례 채택의 핵심 제약** — URL이 죽었거나 인증 필요한 사례는 배제.
3. **사례 A/B는 서로 다른 도메인** — 같은 회사/제품에서 두 사례를 뽑지 않는다 (예: Apple만 두 개 ❌).
4. **물리/추상 메타포가 명확해야 한다** — "이 패턴이 어떤 현실 법칙을 시뮬레이션하는가"가 한 줄로 설명되어야 한다.
5. **본문이 7개 sub-topic이라면 새 패턴도 5~7개** — 본문보다 양을 줄이지 않는다.

## 자주 묻는 질문

**Q. 기존 `_리서치.md` 파일이 있으면?**
A. 덮어쓰지 않는다. `_새로운 패턴 리서치.md`로 별도 저장. 두 문서는 다른 접근(본문 따라가기 vs 본문 너머)이므로 공존한다.

**Q. 본문 주제와 우연히 겹치는 패턴이 떠오르면?**
A. 배제한다. 새 패턴은 반드시 본문 NOTE에 명시되지 않은 메타포여야 한다.

**Q. URL 검증이 실패하면?**
A. WebSearch로 대체 사례를 찾는다. 사례 A/B 중 하나만 살아있어도 그 패턴은 유지하되, B를 새로 채워 넣는다.

**Q. 장 제목에서 키워드를 못 찾으면?**
A. 본문 첫 문장과 NOTE 박스를 함께 읽어 메타 주제를 추론한다. 그래도 불명확하면 사용자에게 한 줄로 묻는다.
